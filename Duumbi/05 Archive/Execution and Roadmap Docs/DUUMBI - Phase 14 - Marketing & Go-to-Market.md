## Domain Strategy

| Domain | Purpose | Status |
|---|---|---|
| `duumbi.dev` | Compiler platform — marketing, docs, registry | To register (Namecheap) |
| `duumbi.com` | Reserved — previously used for real estate platform prototype (2025) | Owned, redirect to duumbi.dev |

**Decision:** `duumbi.dev` is the primary domain for the compiler platform. The `.dev` TLD signals developer tooling (Rust uses `crates.io`, Go uses `pkg.go.dev`). The `.com` domain redirects to `.dev` to capture mistyped traffic.

**DNS Architecture** (from Post-MVP Roadmap, confirmed):

```
Namecheap (registrar: duumbi.dev)
    │
    └── NS records → Azure DNS Zone (duumbi.dev)
        ├── A/CNAME  duumbi.dev              → Landing page (Astro static)
        ├── CNAME    docs.duumbi.dev          → Documentation (mdBook)
        ├── CNAME    registry.duumbi.dev      → Registry Web Frontend
        ├── CNAME    api.registry.duumbi.dev  → Registry API
        ├── CNAME    cdn.duumbi.dev           → Binary Cache CDN
        ├── CNAME    telemetry.duumbi.dev     → Dashboard (PRO+)
        ├── CNAME    status.duumbi.dev        → Uptime monitor
        └── TXT/MX   (SPF, DKIM)             → Email protection

Namecheap (registrar: duumbi.com)
    └── 301 redirect → duumbi.dev
```

## Addendum: Ecosystem Seed Strategy

> Added 2026-03-15.

An empty registry is the single biggest adoption barrier. The solution is not to wait for community contributions but to **pre-seed the registry with high-value stdlib modules** that cover the most common use cases. With FFI (Phase 9a), this becomes tractable — most modules are thin wrappers around proven C libraries, not ground-up reimplementations.

### Seed Stdlib Roadmap

Three tiers: **Essential** (before Phase 14 HN launch), **Important** (within 3 months of launch), **Nice-to-Have** (community-driven).

#### Tier 1: Essential (Ship with Phase 9 — pre-installed by `duumbi init`)

| Module | Wraps | Functionality | FFI? |
|---|---|---|---|
| `@duumbi/stdlib-math` | native Cranelift | abs, max, min, pow, sqrt, mod, bitwise | No |
| `@duumbi/stdlib-string` | native graph | concat, compare, length, find, contains, split, join, trim | No |
| `@duumbi/stdlib-io` | C stdio | print, println, readln, formatted output | Shim |
| `@duumbi/stdlib-file` | C stdio | open, read, write, close, exists | Shim |
| `@duumbi/stdlib-json` | cJSON (vendored) | parse, serialize, get | Shim |
| `@duumbi/stdlib-net` | POSIX sockets | TCP listen, accept, connect, read, write | Shim |
| `@duumbi/stdlib-http` | built on net | HTTP GET, POST, basic request/response | Shim |
| `@duumbi/stdlib-tls` | LibreSSL/OpenSSL | TLS connect, read, write, HTTPS | FFI |
| `@duumbi/stdlib-db` | SQLite (vendored) | open, execute, query, row iteration | FFI |
| `@duumbi/stdlib-server` | poll() event loop | multi-connection server with handler dispatch | Shim |

**10 modules, covering:** math, strings, I/O, files, JSON, networking, TLS, database, server. This is enough for a developer to build a realistic application.

### The FFI Multiplier Effect

FFI (Phase 9a) is the key enabler. Without FFI, every stdlib module must be hand-built in JSON-LD graph — months of work per module. With FFI, a thin wrapper around a C library takes 1–3 days:

| Effort per module | Without FFI | With FFI |
|---|---|---|
| Math (native ops) | 1 week | 1 week (no FFI needed) |
| JSON (cJSON wrapper) | 3 weeks (reimplement parser) | 3 days (wrap cJSON) |
| SQLite | Impossible (too complex) | 1 week (wrap sqlite3.c) |
| TLS | Impossible | 1 week (wrap OpenSSL) |
| Regex | 4+ weeks (reimplement engine) | 3 days (wrap PCRE2) |

**FFI makes the stdlib tractable.** Without it, building 20 modules is a year of work. With it, it's 2–3 months.
